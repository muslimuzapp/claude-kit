0Setup — CLI + local kit

You downloaded the kit; you'll install the ck CLI once, then init your project from the local kit folder using --kit-path. No GitHub, no download, no auth. Do the init once per project.

1

Install the ClaudeKit CLI once, globally

npm install -g claudekit-cli

Meaning: installs the ck command. Needs Node 18+. Verify with ck --version.

2

Init your app from the local kit folder

# go into your existing app (it already has source code)
cd /path/to/ebook-reader

# point ck at the downloaded kit — no network needed
ck init --kit-path "/Users/javokhir/MyFiles/Development/ClaudeKit/ClaudeKit Engineer"

Meaning: --kit-path tells the CLI to install from your local kit directory instead of downloading. It creates the .claude/ folder in your app and leaves your code untouched. Repeat with the same command inside the Habits app. Add -y to skip prompts, --prefix to keep the /ck: namespace.

3

Start Claude Code in the app

claude

Meaning: launches Claude Code with the kit active. The session-init hook detects your project and loads the rules automatically.

4

Generate your CLAUDE.md

/init

Meaning: scans your existing code and writes a CLAUDE.md describing build commands, modules, and conventions. Then hand-add your KMP specifics (module layout, DI, navigation, persistence, expect/actual) — see Golden Rule #4.

Alternatives: ck init --archive ~/downloads/engineer.zip installs from a local zip instead of a folder. Note: since you trimmed this kit, --kit-path installs your slimmed version. If the CLI complains about missing manifest files, point --kit-path at an untrimmed copy of the kit, then delete the unwanted skills from the app's .claude/ afterward. Day-to-day you live in /ck:plan → /ck:cook → /ck:ship.

1How the system works

ClaudeKit is a team of AI specialists coordinated by four layers. Only one layer needs your hands.

LayerWhat it isYou touch it?
CommandsSlash commands you type — the front door to every stage.Yes — you type these
AgentsSpecialist roles the kit delegates to (planner, tester…).No — assigned for you
SkillsPackages of expert knowledge auto-loaded per task.No — auto-activated
HooksBackground scripts that enforce quality gates.No — always on

The 30-second version — for almost every feature you type just three commands:

/ck:plan  "describe the feature"     # think & design
/ck:cook  "implement the plan"       # build & test
/ck:ship  beta                       # commit, review, push, PR

Everything else is optional power tooling for when you explore, design, debug, or explain.

2The Plan → Ship pipeline

Follow top to bottom. Copy the commands as you go.

0

Session start automatic

Nothing to type. Hooks load your CLAUDE.md, the kit rules, and any active plan.

1

Explore the code optional

/ck:scout "how does the reader load and paginate a book"

Meaning: fans out fast parallel searches and reports where the relevant code lives.

2

Plan

/ck:plan "add offline EPUB caching with a 500MB LRU limit"

Meaning: the planner + parallel researcher agents produce a step-by-step plan in ./plans/. No code yet. Use /ck:plan:hard for tricky work, /ck:plan:two to compare approaches.

3

Build

/ck:cook "implement the active plan"

Meaning: fullstack-developer writes the Kotlin/Compose code, ui-ux-designer builds screens, tester runs tests. Use /ck:cook:auto for hands-off.

4

Build UI from a design optional

/ck:design:screenshot ./mockups/reader-settings.png
/ck:design:good "reader settings bottom sheet, Material 3"

Meaning: turns a screenshot or a text description into Compose UI components.

5

Test usually automatic in cook

/ck:fix:test

Meaning: the tester runs unit/integration tests. The session will not finish until tests pass.

6

Review

/ck:code-review

Meaning: the code-reviewer checks quality, edge cases, and security before you ship.

7

Debug when something breaks

/ck:debug
/ck:fix:types   # compile / type errors
/ck:fix:ci      # failing CI pipeline
/ck:fix:ui      # UI glitches

Meaning: the debugger finds the root cause; the fix is applied, then tests re-run.

8

Docs automatic

The docs-manager updates ./docs/ to match your changes.

9

Ship

/ck:ship beta       # to your dev branch
/ck:ship official   # to main (full release: docs + journal)

Meaning: one command runs merge → test → review → commit → push → open PR. Stops only for test failures, critical review issues, or a major version bump.

10

Explain / visualize any time

/ck:preview --diagram "caching data flow"
/ck:preview --html --diff

Meaning: generates diagrams, slide walkthroughs, or a visual diff of your changes.

4The 13 agents — your team

You don't call these directly; commands assign them. Knowing who does what helps you read the output.

planner
Turns a request into a structured, phased plan with TODOs.
researcher
Investigates unknowns — libraries, APIs, best practices. Runs in parallel.
fullstack-developer
Writes the actual code — Kotlin/Compose, logic, data layer.
ui-ux-designer
Designs screens, wireframes, design systems, responsive layouts.
tester
Writes and runs tests, checks coverage, validates behavior.
code-reviewer
Reviews finished code for quality, edge cases, security.
debugger
Investigates bugs and failures, finds the root cause.
code-simplifier
Refactors recent code for clarity without changing behavior.
docs-manager
Keeps ./docs/ accurate and up to date.
git-manager
Stages, writes conventional commits, pushes, opens PRs.
journal-writer
Records what happened after notable fixes or features.
project-manager
Oversees progress, plan status, coordination — handy for you as lead.
brainstormer
Explores ideas and trade-offs with honest pushback.

8Golden rules

  1. Plan before you build. Always /ck:plan first for anything non-trivial — it prevents rework.
  2. Let tests gate you. Don't force a ship past red tests; run /ck:fix:test until green.
  3. One feature, one branch, one /ck:ship. Keeps history clean.
  4. Teach the kit your stack once. A short CLAUDE.md per app (KMP module layout, DI, navigation, persistence, expect/actual) makes every command smarter.
  5. Use /ck:preview --diagram whenever you want to see how a piece fits before committing to it.
Core loop to remember: /ck:plan → /ck:cook → /ck:ship. Everything else is there when you need it.
